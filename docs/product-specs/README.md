# product-specs

**What** we're building and **why** — from the product/user point of view. No
implementation detail.

A spec answers: who is this for, what problem does it solve, what does success
look like, what's in and out of scope.

- **Filename**: `YYYY-MM-DD-short-description.md`.
- Keep it implementation-agnostic — the _how_ belongs in `design-docs/`.
- A design doc should be traceable back to a spec (or an explicit "no spec —
  direct request" note).
